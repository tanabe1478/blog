# AGENTS.md

このリポジトリで AI agent が作業するときの運用ルールです。

## 基本方針

- 応答は基本的に日本語で行う。
- このリポジトリは、ユーザーが AI と一緒に学びながら育てるためのものです。
- 実装をブラックボックス化しない。何を、なぜ、どこに変更するのかを分かる粒度で説明する。
- 大きな変更を一度に入れず、理解・レビューしやすい小さなステップに分ける。

## コミット単位

コミットは、ユーザーが学習しながら追える範囲に小さくする。

望ましい単位:

- 設定ファイルを1つ追加する
- 小さな機能の土台だけを追加する
- 既存コードの小さなリファクタだけを行う
- テストを先に追加する
- テストを通すための最小実装を追加する
- ドキュメントだけを更新する

避ける単位:

- 複数の機能を1コミットにまとめる
- リファクタと機能追加を同じコミットに混ぜる
- テスト追加、実装、UI変更、ドキュメント更新をまとめて入れる
- 自動整形による大量差分を機能変更と混ぜる

## コミットメッセージ

コミットメッセージは丁寧に書く。単に `update` や `fix` で済ませない。

形式の目安:

```text
<type>: <何をしたか>

<なぜ必要だったか>
<主な変更点>
<確認したこと>
```

例:

```text
docs: add agent-browser visual check guide

ブログ統合前の見た目を軽く確認できるようにするため、
agent-browser で確認するページと観点を docs に整理した。

まだ自動 screenshot test や CI 統合は行っていない。
確認: docs の手順を読み、対象ページが分かることを確認
```

よく使う type:

- `feat:` 機能追加
- `fix:` 不具合修正
- `refactor:` 振る舞いを変えない整理
- `test:` テスト追加・修正
- `docs:` ドキュメント更新
- `chore:` 設定や補助作業

## 作業の進め方

1. 変更前に、現在の未コミット差分を確認する。
2. 既存の未コミット変更がある場合は、勝手に混ぜない。
3. これから行う変更を短く説明する。
4. 小さく実装する。
5. 可能なら小さなテストまたは確認コマンドを実行する。
6. 変更内容と確認結果を説明する。
7. コミットする場合は、ユーザーが内容を理解できるメッセージにする。

## 学習しやすさのための説明

大きめの実装では、以下を説明する。

- 変更したファイル
- そのファイルの役割
- 今回の変更理由
- 代替案があった場合は、なぜ採用しなかったか
- 次に読むと理解が深まる箇所

コードを追加するときは、必要に応じて短いコメントやドキュメントを残す。ただし、明らかな処理に過剰なコメントは入れない。

## ブログ統合・CMS・extension 方針

- ブログ固有の機能は Markmesh 本体へ直接入れず、extension として設計する。
- Markmesh 本体へ追加する場合は、一般ユーザーにも有用な extension API / capability / task runner として抽象化する。
- 画像アップロードは Gyazo API を前提にする。
- Gyazo token などの secret は repository に保存しない。安全な secret storage を使う設計にする。
- 見た目の確認は agent-browser を主に使う。自動 screenshot test は必要になった段階で、小さく再導入を検討する。

## 注意

- `git commit --no-verify` は使わない。
- 既存の未コミット変更を勝手に戻さない。
- ユーザーの学習機会を奪うような巨大な一括変更をしない。
